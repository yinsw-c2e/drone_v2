# 无人机货物吊运智慧服务平台产品说明书

> 用途：本说明书面向 Google Stitch 或其他 UI 重设计工具，帮助设计模型理解产品定位、用户角色、业务流程、页面矩阵和重设计目标。当前实现为 uni-app / Vue3 / TypeScript 小程序 MVP，截图位于 `docs/google-stitch/screenshots/`。

## 1. 产品定位

本应用是面向低空经济场景的“无人机货物吊运智慧服务平台”。它服务于业主、飞手、机主和运营管理方，覆盖从发起吊运需求、智能匹配运力、确认方案、空域与合规检查、飞行监控、卸货、结算分账到评价与理赔的完整 MVP 闭环。

产品应呈现为专业、可信、可操作的低空物流调度系统，而不是营销页或普通表单小程序。界面需要体现航线、空域、运力、飞行状态、合规、风险、结算等行业核心对象。

## 2. 目标用户

| 用户 | 目标 | 核心任务 |
| --- | --- | --- |
| 业主 / 发货方 | 快速、安全地发起吊运订单并追踪履约 | 发单、选择起终点、填写货物信息、选择保险/支付、确认匹配方案、监控飞行、结算评价、发起理赔 |
| 飞手 | 接收任务并安全执行飞行作业 | 查看派单、接单、安检、空域/起飞/装卸状态推进、监控遥测、应急处置、查看收益 |
| 机主 | 管理无人机资产与可用运力 | 设备认证、三者险/维护/适航状态管理、投放/撤回运力、查看分账钱包 |
| 管理后台 / 运营方 | 管控平台订单、认证、风控与报表 | 审核认证、订单流转、风险处置、理赔推进、查看审计日志、跑通演示流程 |

## 3. 当前 MVP 能力边界

当前 MVP 是可演示的前端闭环，不代表正式生产验收完成：

- 数据层为前端本地存储、seed 数据和 reactive repo，不是服务端数据库。
- 支付、空域、保险、征信、真机无人机 SDK 均为接口预留和演示返回，不是真实第三方接入。
- 地图选点目前是 H5 / 小程序内的交互式选点层，可接 `uni.chooseLocation`，真实地图瓦片、地址搜索和逆地理编码需要接入地图 Key。
- 管理后台的“跑通”是端到端演示入口，用于生成/推进一条演示流程，不应设计成正式生产按钮。
- 真正生产能力，如 UOM 空域系统、真实清分、保险 API、征信、人脸实名、短信推送、真机视频流、等保三级和高并发能力，属于后续生产化路线。

## 4. 端到端主流程

1. 用户登录并选择角色。
2. 业主进入指挥台，发起吊运需求。
3. 发单页填写货物类型、重量、货值、体积、照片、起终点、时间模式、时效、预算、支付方式和保险意愿。
4. 系统进入智能匹配，按策略推荐合规运力，展示飞手、设备、报价、ETA、距离、保险与合规保障。
5. 业主确认方案后进入订单追踪。
6. 飞手在接单大厅或任务页接收任务，完成空域、安检、装货、飞行、卸货等状态推进。
7. 系统生成结算与分账，飞手/机主钱包显示流水与提现。
8. 业主评价服务；如发生货损或纠纷，可进入保险理赔流程。
9. 管理后台查看订单、认证、风控、报表和审计日志。

## 6. 视觉与交互重设计目标

### 6.1 产品气质

- 专业低空物流调度系统。
- 暗色“低空指挥中心”风格，但不要滥用假航线头图。
- 数据和状态驱动，航线、空域、运力、飞行、风险和结算是视觉主角。
- 克制、清晰、可执行，不要像模板 demo、组件库示例或营销页。

### 6.2 移动端设计要求

- 390 / 430 宽度下无横向溢出、无按钮竖排、无文字随意断行。
- 每屏只保留一个主行动，次行动降级。
- 固定底栏必须为内容预留安全空间，不遮挡最后一块内容。
- 重要信息按“状态 / 下一步 / 责任方 / 金额或风险 / 操作”组织。
- 所有 KPI、标签、状态、枚举都必须是中文业务文案。
- 空态、终态和阻断态必须告诉用户下一步。

### 6.3 后台桌面设计要求

- 1280 桌面宽度下应像真实运营后台，而不是移动卡片拉宽。
- 需要侧栏/顶部态势/横向指标/表格式队列/报表/审计日志。
- 认证、订单、理赔、审计需要可扫读的队列结构和对齐的操作列。
- 移动端回落时仍要可读，动作按钮不能单字竖排。

## 7. 已知需要重点重设计的体验问题

这些问题来自当前产品 QA，可作为 Stitch 重设计重点：

- 发单页不应该在每个页面都套航线头图；表单页应以任务信息和地图选点为主。
- 地点选择必须像真实地图选点，而不是三个固定地点列表。
- 货物照片上传应是一个清晰上传入口，不需要重复的“+”和“选择照片”双按钮。
- 首页快捷入口和飞手入口卡片目前视觉仍偏硬，应改成更自然的工具入口或底部功能区。
- KPI 卡片应可点击进入对应功能，如信用详情、通知大厅、任务详情。
- 钱包提现需要输入金额/比例选择/确认，不应该固定生成小额提现。
- 保险理赔不能重复点击“报案”生成多个已报案条目；理赔动作区需要更清晰。
- 管理后台的“端到端验收”是演示测试入口，正式设计中应弱化或放入开发/演示工具区。
- 认证页、钱包页等固定底栏页面必须处理底部安全间距。
- 设备与运力卡片需要更像资产管理，不要文字贴边、指标挤压、标签难读。
